Plugin Package Structure & Types

How plugins are structured, what we expose, and how existing plugins migrate

Package Structure Decision

Should CodeGraphy config live in package.json or a separate manifest?

Option A: package.json only

Follow VS Code convention. Everything under a "codegraphy" field.

// package.json
{
  "name": "codegraphy-metrics",
  "displayName": "Code Metrics for CodeGraphy",
  "version": "1.0.0",
  "engines": { "vscode": "^1.85.0" },
  "extensionDependencies": [
    "codegraphy.codegraphy"
  ],
  "main": "./dist/extension.js",
  "contributes": {
    "configuration": {
      // Plugin's own VS Code settings
    }
  },
  "codegraphy": {
    "apiVersion": "^2.0.0",
    "tier": 1,
    "supportedExtensions": [".ts", ".tsx"],
    "capabilities": ["decorations", "events"],
    "rules": [...],
    "fileColors": {...}
  }
}

Pros

  • One file — VS Code convention
  • NPM ecosystem already understands it
  • GitLens, ESLint, etc. all do this

Cons

  • package.json gets large
  • Mixes npm + vscode + codegraphy concerns

Option B: package.json + codegraphy.json

Like Obsidian's manifest.json. Separate file for CodeGraphy-specific config.

// package.json (standard vscode stuff only)
{
  "name": "codegraphy-metrics",
  "extensionDependencies": [
    "codegraphy.codegraphy"
  ],
  "main": "./dist/extension.js"
}

// codegraphy.json (plugin manifest)
{
  "id": "codegraphy-metrics",
  "apiVersion": "^2.0.0",
  "tier": 1,
  "supportedExtensions": [".ts", ".tsx"],
  "capabilities": ["decorations", "events"],
  "rules": [...],
  "fileColors": {...},
  "webviewContributions": {
    "scripts": ["dist/webview.js"],
    "styles": ["dist/webview.css"]
  }
}

Pros

  • Clean separation of concerns
  • JSON schema validation for codegraphy.json
  • Easier to parse for tooling
  • Obsidian devs will recognize it

Cons

  • Two files to maintain
  • Non-standard for VS Code ecosystem
Recommendation: Option B. The codegraphy.json manifest keeps things clean and lets us provide a JSON schema for validation + autocomplete in editors. Built-in plugins use the same manifest format. We can also publish a codegraphy.schema.json so plugin authors get intellisense when editing their manifest.

Plugin author usage
// npm install -D @codegraphy/plugin-api

import type {
  IPlugin,
  CodeGraphyAPI,
  IConnection,
  NodeDecoration,
  EventPayloads,
} from '@codegraphy/plugin-api';

import type {
  CodeGraphyWebviewAPI,
  NodeRenderFn,
} from '@codegraphy/plugin-api/webview';

// Full autocomplete + type safety
const plugin: IPlugin = {
  id: 'my-plugin',
  apiVersion: '^2.0.0',

  onLoad(api: CodeGraphyAPI) {
    // api. ← full intellisense
    api.on('graph:nodeClick', (e) => {
      // e is typed as EventPayloads['graph:nodeClick']
      console.log(e.node.id);
    });

    api.decorateNode('file.ts', {
      badge: { text: '3', color: 'red' },
      // ← autocomplete shows all decoration options
    });
  }
};

Key Exposed Types

CodeGraphyAPI
// The main API plugins receive in onLoad
{
  version: string;

  // Events
  on(event, handler): Disposable;
  once(event, handler): Disposable;
  off(event, handler): void;

  // Decorations
  decorateNode(id, decoration): Disposable;
  decorateEdge(id, decoration): Disposable;
  clearDecorations(pluginId?): void;

  // Graph queries
  getGraph(): IGraphData;
  getNode(id): IGraphNode | null;
  getNeighbors(id): IGraphNode[];
  getEdgesFor(id): IGraphEdge[];

  // Registration
  registerView(view): Disposable;
  registerCommand(cmd): Disposable;
  registerContextMenuItem(item): Disposable;

  // Webview communication (Tier 2)
  sendToWebview(msg): void;
  onWebviewMessage(handler): Disposable;

  // Utilities
  getWorkspaceRoot(): string;
  log(level, ...args): void;
}
NodeDecoration
// All structured decoration options
{
  badge?: {
    text: string;
    color?: string;
    bgColor?: string;
    position?: 'top-right' | 'top-left'
              | 'bottom-right' | 'bottom-left';
    tooltip?: string;
  };
  border?: {
    color: string;
    width?: number;
    style?: 'solid' | 'dashed' | 'dotted';
  };
  tooltip?: {
    sections: TooltipSection[];
  };
  label?: {
    text?: string;       // override display name
    sublabel?: string;   // secondary text below
    color?: string;
  };
  size?: {
    scale?: number;      // multiplier (1.0 = default)
  };
  opacity?: number;      // 0.0 - 1.0
  color?: string;        // override node color
  icon?: string;         // codicon name
  group?: string;        // visual grouping id
  priority?: number;     // when multiple plugins decorate same node
}
EventPayloads (type map)
// Typed event map for full intellisense
interface EventPayloads {
  'graph:nodeClick': {
    node: IGraphNode;
    event: { x: number; y: number };
  };
  'graph:selectionChanged': {
    nodes: IGraphNode[];
    edges: IGraphEdge[];
  };
  'analysis:completed': {
    graph: IGraphData;
    duration: number;
  };
  'workspace:fileChanged': {
    filePath: string;
  };
  // ... all 30 events fully typed
}

// Usage: api.on() is generic
api.on<'graph:nodeClick'>(
  'graph:nodeClick',
  (e) => e.node.id // ← typed!
);

  };

What a third-party plugin repo looks like

├── tsconfig.json
├── src/
│   ├── extension.ts         // activate() → registerPlugin()

│   ├── extension/

│   ├── extension.js

Migrating Existing Built-in Plugins

5 built-in plugins eat our own API — serving as reference implementations

Plugin Migration Scope New Capabilities Reference For
TypeScript Largest — 4 rules, 23 colors, AST parsing, tsconfig resolution Add tooltips (show import type), config file badges Gold standard example
Python Small — 2 rules, regex-based detection Add venv detection, __init__.py grouping Simple plugin template
C# Medium — namespace resolution, cross-file types Add namespace grouping view preAnalyze + views
GDScript Medium — preAnalyze for class_name map, res:// paths Add scene tree grouping, resource badges preAnalyze pattern
Markdown Small — wikilink detection, file index Add backlink count badges, tag-based grouping Obsidian-style features
Key principle: Built-in plugins use the exact same API as third-party plugins.

They ship bundled with the extension but register through registerPlugin() like everyone else. This guarantees we dogfood our own API, and if a built-in plugin can't do something, we know the API is missing a capability. The TypeScript plugin becomes our "gold standard" reference that plugin authors can study.

Developer Experience Tooling

@codegraphy/plugin-api
NPM package with all types. devDependency for plugins. Versioned in lockstep with core API version.
codegraphy.schema.json
JSON Schema for codegraphy.json manifest. Gives autocomplete + validation in any editor.
create-codegraphy-plugin
CLI scaffolder. npx create-codegraphy-plugin → picks Tier 1 or 2, generates project with TypeScript, esbuild, tests.
